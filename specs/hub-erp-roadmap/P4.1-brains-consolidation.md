# P4.1 — AI Brains consolidation (nav merge + managed agents + friendly sources)

**Repo:** minion_hub (`dev`). Hub-only — no gateway repo changes (agent provisioning reuses the existing hub→gateway `agents.create`/`config.patch` path).
**Goal:** the sidebar's two overlapping items — "AI Brains" (`/agents?archetype=brain` roster filter) and "Knowledge Bases" (`/brains` module) — become ONE "AI Brains" module at `/brains` with SideNav subtabs. Every brain can have a managing agent provisioned from a single org-level template (reconfigure template → all brain agents updated). Data sources get a friendly unified picker covering internal (db modules) and external (URL, upload; Drive/S3 visible as coming-soon).

## Design decisions (locked)

1. Route stays `/brains`. Module display name becomes **"AI Brains"** everywhere. The `nav_knowledgeBases` sidebar item is retired; the `archetypeItem("brain", …)` sidebar item is removed (the `?archetype=brain` URL keeps working — just no nav entry). The `brain` archetype itself is unchanged and is the "class" brain agents use.
2. Subtabs = new `BrainsNav.svelte` (SideNav, FinanceNav pattern) mounted in a new `/brains/+layout.svelte`:
   - **Brains** `/brains` — existing card grid (unchanged behavior).
   - **Agents** `/brains/agents` — brain-archetype agents: which brain each manages, status, link to `/agents?archetype=brain` roster detail.
   - **Template** `/brains/template` — the org's Brain Agent Template editor. Gated `brains:manage` (hide link via canViewPath-style check; server-guard the route).
   - Per-brain detail keeps its inner Tabs; "Documents" tab is renamed **"Sources"**; a new **"Agent"** tab is added.
3. Managing-agent model:
   - New table `brain_agent_templates` — ONE row per org: `org_id` (unique), `name_prefix` (default "Brain"), `emoji`, `model` (nullable = gateway default), `instructions` (text, the shared system prompt; supports `{{brain_name}}`/`{{brain_description}}` placeholders), timestamps.
   - New nullable column `brains.agent_id` — the gateway agentId of the brain's managing agent, pattern `brain-<brainUuid>`.
   - Provisioning (opt-in, NOT automatic on create): "Enable agent" on the brain's Agent tab → `POST /api/brains/[id]/agent` (gate `brains:manage`) → creates gateway agent `brain-<uuid>` with archetype `brain`, identity from template (placeholders rendered), stores `agent_id`, and inserts a `brain_access` row (`agent`, `brain-<uuid>`, `write`). Follow the personal-agent create endpoint's privileged gateway-call shape exactly.
   - Template save → `PUT /api/brains/template` re-renders + `config.patch`es EVERY brain agent in the org (list = `brains where agent_id is not null`). This is the "reconfigure once, all impacted" guarantee. Fan-out failures: collect per-agent errors, return them, don't abort the batch.
   - Disable agent → delete gateway agent + null `agent_id` + remove its access row. Audit all of it (`recordAudit`, refType `brain`).
   - `resolveAssistantPrincipal` only resolves `personal-<uuid>` agents; brain agents (`brain-<uuid>`) do NOT get hub API principals in this phase — they operate gateway-side only. Leave a `// P4.2:` comment where a brain-agent principal resolver would slot in.
4. Sources UX (any technical level):
   - One **"Add source"** button on the Sources tab → dialog with option cards: ✍️ Write a note · 🔗 Add a link · 📄 Upload a file · 🧩 Connect app data · (disabled, "Coming soon") Google Drive · S3 bucket. Each card = one plain-language sentence.
   - **Upload v1 = small text files inline**: accept `.md .txt .csv`, ≤ 1 MB, read client-side (FileReader), submit as `source_type:'upload'`, content into `content_md`, `source_ref` = filename. Service: implement the `upload` branch of `loadDocumentContent` (content_md as-is, like note). No B2, no binary formats (PDF/DOCX deferred — the picker card says "Text files for now").
   - **Connect app data = module_ref catalog**: replace the hardcoded fin_products branch with a `MODULE_SOURCES` registry `{ key, label(), description(), requiredPerm, render(orgId) → markdown }`. v1 entries: `fin_products` (existing renderer), `crm_contacts` (parties spine: name, stage, tags, lifecycle), `stk_items` (item, uom, qty on hand from bins). Picker lists registry entries the user can view (check requiredPerm); each renders to markdown then goes through the normal chunk/embed path. Keep the existing `// P2 hub_events` re-ingest comment.
   - Status language: map pending/ingesting → "Learning…", ready → "Ready", failed → "Needs attention" (i18n, both locales).
5. i18n: module strings move to/stay under `brains_*`; `nav_brains` stays "AI Brains"; retire `nav_knowledgeBases` usage (key may remain in messages files if removal churns; just stop referencing it). en + es + `bun run i18n:compile`.
6. RBAC (REQUIRED, per hub CLAUDE.md checklist): new routes `/brains/agents` + `/brains/template` are under `/brains` → covered by the existing route view guard, but `/brains/template` and the agent/template APIs additionally require `brains:manage` (use `requireOrgCapability`). New API prefixes under `/api/brains` are already in `apiWriteCapability`'s table — verify, don't assume.

## Migration (agent WRITES the file only; orchestrator applies via Supabase MCP)

```sql
create table if not exists brain_agent_templates (
  id uuid primary key default gen_random_uuid(),
  org_id text not null unique,
  name_prefix text not null default 'Brain',
  emoji text,
  model text,
  instructions text not null default '',
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
alter table brains add column if not exists agent_id text;
```
