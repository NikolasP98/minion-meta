### B3 — `minion-core` migration strategy (DECIDED IN TASK 1 DISCOVERY)

**Problem:** The existing shared-secrets project lives at UUID `dd71e710-4e1a-48f6-afea-5502bae5a574` (the "ai-providers" project referenced by `infisical-dev.sh`). After 02-08 deprecates `infisical-dev.sh`, this UUID becomes orphaned unless secrets are migrated to a `minion-core`-named project.

**Option chosen (locked):** **Option (a) — RENAME the existing UUID project's slug to `minion-core`.**

Why: It preserves every existing secret (zero data migration risk), is a single dashboard/API operation, and leaves the UUID reference in `infisical-dev.sh`'s deprecation shim still resolvable (though the shim now just prints a warning and exits — UUID never dereferenced in runtime paths).

**Fallback paths (used ONLY if DISCOVERY Task 1 reports Option (a) unsupported):**

- **Option (b)** Create new `minion-core` + migrate secrets:
  ```bash
  DOMAIN=http://100.80.222.29:8080/api
  # Export secrets from old UUID project
  infisical secrets --projectId dd71e710-4e1a-48f6-afea-5502bae5a574 --env dev --domain "$DOMAIN" -o dotenv --silent > /tmp/minion-core-export.env
  OLD_COUNT=$(grep -cE "^[A-Z_]+=" /tmp/minion-core-export.env)
  # Import into new minion-core project
  # (infisical secrets set does not accept --file; loop:)
  while IFS='=' read -r k v; do
    [ -z "$k" ] && continue
    case "$k" in \#*) continue ;; esac
    infisical secrets set "$k=$v" --projectSlug minion-core --env dev --domain "$DOMAIN" >/dev/null 2>&1
  done < /tmp/minion-core-export.env
  # Verify count match
  NEW_COUNT=$(infisical secrets --projectSlug minion-core --env dev --domain "$DOMAIN" -o dotenv --silent | grep -cE "^[A-Z_]+=")
  [ "$OLD_COUNT" = "$NEW_COUNT" ] || { echo "FAIL: secret count drift $OLD_COUNT → $NEW_COUNT"; exit 1; }
  shred -u /tmp/minion-core-export.env
  ```
