<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: AUTH-04 shared-session continuity smoke test (hub to site)</name>
  <files>(no files modified — curl-based verification only; outputs captured for Task 3)</files>
  <what-built>Both staging apps run `createAuth()` with the same secret, same DB, and identical JWT audience. A user signing up on hub should have a session recognized by site via the shared `session` table in the shared Turso DB.</what-built>
  <how-to-verify>
    Set shell variables from Task 1's reply, then run each step and record all output for Task 3:

    ```bash
    HUB=https://<hub-staging-url>
    SITE=https://<site-staging-url>
    EMAIL="phase6-check-$(date +%s)@test.minion.example"
    PASSWORD="Phase6TestPassword!"
    ```

    **Step 1 — Sign up on hub:**
    ```bash
    curl -i -c hub-cookies.txt -X POST "$HUB/api/auth/sign-up/email" \
      -H 'content-type: application/json' \
      -d "{\"email\":\"$EMAIL\",\"password\":\"$PASSWORD\",\"name\":\"Phase6 Hub User\"}"
    ```
    Expect: HTTP 200, `set-cookie` header with a `better-auth.session_token` cookie.

    **Step 2 — Verify session on hub:**
    ```bash
    curl -s -b hub-cookies.txt "$HUB/api/auth/session" | jq .
    ```
    Expect: JSON with `user.email === "$EMAIL"` and a `session.id`.

    **Step 3 — Attempt session on site using the same cookie jar:**
    ```bash
    curl -s -b hub-cookies.txt "$SITE/api/auth/session" | jq .
    ```
    Expected outcomes:
    - PASS: Returns JSON with `user.email === "$EMAIL"`.
    - PARTIAL (cross-subdomain cookies not enabled): Returns null or 401. In curl, `-b` does not enforce browser domain scoping — a null response here means either the secret is mismatched OR the session row is not in the expected DB. A cookie-domain mismatch would manifest differently in a real browser. Document the raw curl outcome and the `Set-Cookie Domain` attribute from hub-cookies.txt.

    **Step 4 — JWKS parity check:**
    ```bash
    curl -s "$HUB/api/auth/jwks"  | jq -S '.keys | map(.kid) | sort' > hub-kids.json
    curl -s "$SITE/api/auth/jwks" | jq -S '.keys | map(.kid) | sort' > site-kids.json
    diff hub-kids.json site-kids.json
    ```
    Expect: diff output is empty — both apps serve identical JWKS keys from the shared `jwks` table.

    **Step 5 — JWT claim inspection:**
    ```bash
    curl -s -b hub-cookies.txt "$HUB/api/auth/token" | jq -r '.token' > hub-jwt.txt
    cut -d. -f2 hub-jwt.txt | base64 -d 2>/dev/null | jq '{iss, aud, exp}'
    ```
    Expect: `aud === "openclaw-gateway"`, `iss === hub staging URL`, `exp` is present.

    **Step 6 — Reverse direction (sign in on site, check hub):**
    ```bash
    curl -i -c site-cookies.txt -X POST "$SITE/api/auth/sign-in/email" \
      -H 'content-type: application/json' \
      -d "{\"email\":\"$EMAIL\",\"password\":\"$PASSWORD\"}"
    curl -s -b site-cookies.txt "$SITE/api/auth/session" | jq .user.email
    curl -s -b site-cookies.txt "$HUB/api/auth/session"  | jq .user.email
    ```
    Expect: both return `"$EMAIL"`.

    **Step 7 — Cookie domain inspection (for cross-subdomain topology):**
    If hub and site are on different subdomains of a shared root in staging:
    ```bash
    grep 'better-auth' hub-cookies.txt
    ```
    Check the `Domain` column: if host-only (e.g., `hub.staging.example.com`), browser-based cross-subdomain sharing requires `crossSubDomainCookies` config — document as a finding for Plan 06-05.

    Record every step's exact output. Redact session tokens to first 8 characters.
  </how-to-verify>
