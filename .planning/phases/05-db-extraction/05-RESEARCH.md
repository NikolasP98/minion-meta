## Summary

The hub schema is 38 TypeScript source files (37 domain files + 1 `auth/` subdirectory with its own `index.ts`) exporting **56 table constants** total. The dependency graph inside the schema is a clean DAG: virtually all tables import from `./auth` (for `organization` or `user` references) and many import from `./servers`. No circular dependencies exist. Zero SvelteKit virtual module imports (`$env`, `$lib`, `$server`) appear in any schema file ŌĆö they are plain TypeScript that can be extracted verbatim into a standalone npm package.

The site has a separate, smaller schema (12 exported tables in 5 files) that is a **subset and is currently out of sync** with the hub: the site's `user` table lacks the `personalAgentId` column that hub added, and the site is missing the OIDC plugin tables (`oauthApplication`, `oauthAccessToken`, `oauthConsent`). This drift must be resolved during extraction by using hub's schema as the single source of truth and having the site consume that.

The hub uses `db:push` as its primary migration tool (not `db:migrate`). The migration journal covers entries 0-7 but 12 SQL files (0000-0011) exist in `drizzle/`. Files 0008-0011 were generated by `drizzle-kit generate` but are not tracked in the journal ŌĆö they were applied to the live database via `db:push`, not `db:migrate`. This means the meta-repo migration runner should also use `db:push` (drizzle-kit push) as the canonical migration mechanism, not `db:migrate`.

**Primary recommendation:** Copy all 38 schema files verbatim into `packages/db/src/schema/` preserving directory structure (including `auth/`), build with `tsc`, publish as `@minion-stack/db@0.1.0`. Hub and site update their `drizzle.config.ts`/client imports. Meta-repo adds a `drizzle.config.ts` that references `packages/db/src/schema/**/*.ts` for the migration-ownership cutover.

## Drizzle Migration Mechanics

### db:push vs db:migrate

**Hub uses `db:push` as primary migration tool.** Evidence: [VERIFIED: codebase inspection]

- Migration journal (`drizzle/meta/_journal.json`) tracks only entries 0-7
- 12 SQL files exist (0000-0011) ŌĆö files 0008-0011 are generated SQL but NOT in the journal
- This pattern: `db:generate` creates the SQL file, but `db:push` applies schema directly to DB without updating the journal
- The `drizzle.config.ts` uses `dialect: 'turso'` which supports both push and migrate

**Implication for meta-repo migration runner:** The meta-repo `db:push` command applies schema directly from `@minion-stack/db` source files. No `db:migrate` approach needed. The migration "runner helper" in `@minion-stack/db` is a `drizzle.config.ts` template or a script that calls `drizzle-kit push`.

### drizzle.config.ts for Meta-Repo

The meta-repo needs its own `drizzle.config.ts` at root:
```typescript
import { defineConfig } from 'drizzle-kit';
export default defineConfig({
  dialect: 'turso',
  schema: ['./packages/db/src/schema/**/*.ts'],
  out: './packages/db/drizzle',
  dbCredentials: {
    url: process.env.TURSO_DB_URL ?? 'file:./minion_hub/data/minion_hub.db',
    authToken: process.env.TURSO_DB_AUTH_TOKEN,
  },
});
```

The `schema` glob points at the workspace source ŌĆö no need to reference node_modules dist. [ASSUMED: drizzle-kit supports workspace-local source paths]

### Hub's drizzle.config.ts After Cutover

**Step 1 (hub retains runner):** Hub `drizzle.config.ts` schema glob changes from `'./src/server/db/schema/**/*.ts'` to reference `node_modules/@minion-stack/db/src/schema/**/*.ts` OR the hub's drizzle.config.ts can import from the dist. The safest approach: hub's `drizzle.config.ts` schema changes to `['../packages/db/src/schema/**/*.ts']` using a relative path (since meta-repo root is parent of minion_hub). However, since hub is a separate git repo, it cannot reference the meta-repo's workspace directly. [VERIFIED: directory structure inspection]

**Recommended approach for Step 1:** Hub `drizzle.config.ts` schema path points at `./node_modules/@minion-stack/db/dist` ŌĆö but `drizzle-kit` needs TypeScript source for schema analysis, not dist JS. The correct pattern is: hub's `drizzle.config.ts` uses `schema: ['node_modules/@minion-stack/db/src/**/*.ts']` ŌĆö but this requires `@minion-stack/db` to include `src/` in its `files` array (npm package). [ASSUMED: drizzle-kit can read .ts from node_modules when schema glob matches]

**Alternative approach for Step 1:** Keep hub's `drizzle.config.ts` pointing at its local `./src/server/db/schema/` which is now a thin wrapper that re-exports everything from `@minion-stack/db`. This avoids the node_modules glob issue entirely.

**Step 2 (meta-repo owns):** Hub's `db:push` script removed from `package.json`. Meta-repo's `db:push` script in root `package.json` becomes the canonical command.

### Migration Files

The existing 12 SQL files in `minion_hub/drizzle/` should move to `packages/db/drizzle/` as part of the extraction ŌĆö they are schema history, not hub-specific.

## LibSQL/Turso Client Setup

### Current Connection Pattern (Hub and Site)

Both hub and site use the **identical client pattern** [VERIFIED: both client.ts files]:

```typescript
import { drizzle } from 'drizzle-orm/libsql';
import { createClient } from '@libsql/client';
import { env } from '$env/dynamic/private';  // SvelteKit virtual ŌĆö stays in consumer
import * as schema from './schema';            // ŌåÆ will change to '@minion-stack/db'

let _db: ReturnType<typeof createDrizzle> | null = null;

function createDrizzle() {
  const url = env.TURSO_DB_URL ?? 'file:./data/minion_hub.db';
  const authToken = env.TURSO_DB_AUTH_TOKEN;
  const client = createClient({ url, authToken });
  return drizzle(client, { schema: allSchema });
}

export function getDb() {
  if (!_db) _db = createDrizzle();
  return _db;
}
```

### What Moves to @minion-stack/db vs Stays in Consumer

| Element | Location After Phase 5 | Reason |
|---------|----------------------|--------|
| Schema table definitions | `@minion-stack/db` | Core extraction goal |
| Relations definitions | `@minion-stack/db` | Drizzle requires relations co-located with schema |
| `createClient()` / `drizzle()` call | Stays in hub's `client.ts` + site's `client.ts` | Uses `$env/dynamic/private` (SvelteKit virtual module) |
| `getDb()` singleton | Stays in hub's `client.ts` + site's `client.ts` | Runtime, not schema concern |
| `drizzle.config.ts` | Meta-repo root (Step 2) / hub (Step 1) | Migration runner config |
| `drizzle/` migrations dir | `packages/db/drizzle/` | Schema history belongs with schema |
| `utils.ts` (newId, nowMs) | Recommendation: move to `@minion-stack/db` as `./utils` export | Used by DB layer, not SvelteKit-specific |

**Recommendation: No.** The hub and site already have working client factories. A `createClient()` factory in `@minion-stack/db` would need to solve the `$env` problem (SvelteKit virtual modules) which is Phase 6 territory (auth factory). Keep `@minion-stack/db` as a schema-and-config-only package for Phase 5.

- `TURSO_DB_URL` ŌĆö LibSQL connection string (Turso production, SQLite local)
- `TURSO_DB_AUTH_TOKEN` ŌĆö optional for SQLite local, required for Turso

## Packaging Pitfalls

### Pitfall 1: SvelteKit Virtual Modules in Schema Files

**Status: Non-issue.** [VERIFIED: grep confirmed zero `$env`/`$lib`/`$server` imports in any schema file or relations.ts]

Schema files only import from `drizzle-orm/sqlite-core`. The `$env` usage is confined to `client.ts` in each consumer ŌĆö it stays there.

### Pitfall 2: drizzle-kit Needs TypeScript Source, Not Compiled dist

drizzle-kit reads TypeScript source to generate schema snapshots. When hub or meta-repo runs `drizzle-kit push`, the `schema` key in `drizzle.config.ts` must point at `.ts` source files, not compiled `.js`. [ASSUMED: based on Drizzle documentation behavior]

**Solutions:**
- Option A (recommended): `@minion-stack/db` includes `src/` in its npm `files` array ŌåÆ hub references `node_modules/@minion-stack/db/src/**/*.ts`
- Option B: Hub keeps thin re-export wrappers in `./src/server/db/schema/` that forward to `@minion-stack/db` ŌĆö drizzle-kit reads hub's local re-exports, which re-export from node_modules (but drizzle-kit may not follow re-exports through node_modules)
- Option C (Step 1 simplification): Hub's `drizzle.config.ts` continues pointing at `./src/server/db/schema/**/*.ts` which becomes re-export stubs. Drizzle-kit follows the imports through to node_modules.

### Pitfall 3: Auth Subdirectory Import Resolution

Hub schema files import `from './auth'` ŌĆö TypeScript resolves this as the `auth/` directory via `auth/index.ts`. When copying to `packages/db/src/schema/`, this directory structure must be preserved exactly. If the auth tables were flattened to `auth.ts`, all 20+ files that import `from './auth'` would need updating. [VERIFIED: inspection of import patterns]

**Recommendation:** Copy `auth/index.ts` verbatim to `packages/db/src/schema/auth/index.ts`. Zero import changes in other schema files.

### Pitfall 4: peer deps for drizzle-orm

`drizzle-orm` is a **peer dependency** of the schema package. Schema files use types from `drizzle-orm/sqlite-core` ŌĆö the package needs `drizzle-orm` to build and consumers must have it installed. Hub and site already have `drizzle-orm: ^0.45.1`. [VERIFIED: both package.json files]

**Package structure:**
```json
{
  "peerDependencies": {
    "drizzle-orm": ">=0.45.0"
  },
  "devDependencies": {
    "drizzle-orm": "^0.45.1"  // for build
  }
}
```

### Pitfall 5: `files` Array Must Include `src/`

For drizzle-kit to read the TypeScript source from node_modules, `packages/db/package.json` must include `"src"` in the `files` array. The `packages/shared/package.json` only includes `["dist", "README.md"]` ŌĆö this pattern must be extended for `@minion-stack/db` to also include `"src"`.

### Pitfall 6: pnpm minimumReleaseAge for @minion-stack/* in minion/

The `minion/` subproject has `pnpm.minimumReleaseAgeExclude` for `@minion-stack/tsconfig` and `@minion-stack/lint-config` but NOT for future packages. Once `@minion-stack/db` is published, if `minion/` ever consumes it, it needs to be added to the exclude list. For Phase 5 (hub/site only, bun-based), this is not immediately relevant. [VERIFIED: minion/package.json pnpm config]

### Pitfall 7: Site Schema Drift Will Cause Auth Runtime Errors

The site's local `user` table lacks `personalAgentId`. After switching to `@minion-stack/db`, the site's schema will include `personalAgentId` ŌĆö this column exists in the shared database (hub added it) so no migration is needed. The site's auth and query code must not reference `personalAgentId` (it doesn't, the site doesn't use that feature). This is safe. [VERIFIED: site auth.ts inspection]

### Pitfall 8: `reliability_events` naming discrepancy

The hub schema file is `reliability-events.ts` exporting `reliabilityEvents` (camelCase). Python analysis mistakenly listed `reliability_events` (snake_case) ŌĆö this is a false artifact. The actual export is `reliabilityEvents`. [VERIFIED: file listing and schema inspection]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Schema-to-DB synchronization | Custom migration scripts | `drizzle-kit push` | drizzle-kit handles Turso/SQLite dialect differences, FK pragma, column type coercion |
| Schema snapshot/diffing | Manual diff | `drizzle-kit generate` creates snapshots | Snapshot format is internal to drizzle-kit; hand-rolling breaks the journal |
| ID generation | `crypto.randomUUID()` | `@paralleldrive/cuid2` via existing `utils.ts` | Already in codebase; cuid2 produces URL-safe short IDs unlike UUID |
| Type-safe query builder | Raw SQL | Drizzle relational query API (`db.query.agents.findMany()`) | Relations.ts already defines this; consumers get typed query API for free |

## Two-Step Cutover Risks

### Risk 1: drizzle-kit schema path resolution for node_modules

**What goes wrong:** `drizzle-kit push` with `schema: ['./node_modules/@minion-stack/db/src/**/*.ts']` may fail if drizzle-kit doesn't follow TypeScript imports inside node_modules, or if bun's node_modules layout differs from npm/pnpm.

**Why it happens:** drizzle-kit expects TypeScript source on a local path. Node_modules is typically not treated as project source.

**How to avoid:** Test with hub's `db:push` pointing at `node_modules/@minion-stack/db/src/` immediately after plan 05-01 publish. If it fails, fall back to Option B: hub keeps thin local re-export stubs during Step 1.

**Warning signs:** `drizzle-kit push` exits non-zero with "cannot find module" or "no tables found" error.

### Risk 2: Migration journal gap (entries 0004-0005, 0008-0011)

**What goes wrong:** The migration journal only tracks entries 0-7, but the current schema (after `db:push` applied 0008-0011) is at a higher state. If the meta-repo runs `db:generate` against the full current schema source and compares to the journal, it may attempt to re-apply already-applied changes.

**Why it happens:** Journal was not updated when `db:push` was used after 0007. The snapshots only go up to 0007 (with gaps at 0004, 0005).

**How to avoid:** The meta-repo uses `db:push` (not `db:migrate`), which compares schema to the live database state ŌĆö not to the journal. `db:push` is idempotent against the live DB. [ASSUMED: drizzle-kit push compares to live DB, not to journal]

### Risk 3: Site schema drift causes auth query failures

**What goes wrong:** Site's local `user` table lacked `personalAgentId`. After switching to `@minion-stack/db`, the schema includes this field. If the Drizzle client sends a query that references `personalAgentId` in the site context, it will fail with a column error (the column exists in DB but site code doesn't reference it ŌĆö inverse problem is fine).

**Why it happens:** Not a real risk ŌĆö the site code never queries `personalAgentId`. The column is present in the database (hub added it). Drizzle schema inclusion of extra columns is safe if not queried.

### Risk 4: Hub's seed.ts and auth.ts use `$server/db/schema` path alias

**What goes wrong:** Hub's `src/server/seed.ts` imports `from './db/schema'` and hub's auth imports `from '$server/db/schema'`. These must be updated to `from '@minion-stack/db/schema'` (or `from '@minion-stack/db'`).

**Why it happens:** SvelteKit path alias `$server ŌåÆ src/server/` is used throughout hub. The alias itself is fine; only the import target changes.

**How to avoid:** Grep for all `from.*db/schema` and `from.*db/relations` in both hub and site source trees; update all in plan 05-03.

**Warning signs:** TypeScript compile errors in hub after removing local schema files.

### Risk 5: Better Auth 1.4.19 drizzle adapter passes full schema

**What goes wrong:** `drizzleAdapter(getDb(), { provider: 'sqlite', schema })` ŌĆö if `schema` is a wildcard import from `@minion-stack/db` that includes relations, it may confuse Better Auth's internal schema detection.

**Why it happens:** Hub currently passes `{ ...schema, ...relations }` as the DB schema but passes only `schema` to Better Auth. Post-migration, this distinction must be maintained.

**How to avoid:** Hub's `client.ts` imports `schema` and `relations` separately; passes `allSchema = {...schema, ...relations}` to drizzle, but `schema` only to `drizzleAdapter`. Site does the same.

## Common Pitfalls

### Pitfall 1: drizzle-kit Cannot Read dist/ for Schema Analysis

**What goes wrong:** Setting `schema: './node_modules/@minion-stack/db/dist/**/*.js'` in drizzle.config.ts causes drizzle-kit to fail or produce empty diffs ŌĆö it needs TypeScript, not compiled JS.

**Why it happens:** drizzle-kit's schema parser reads TypeScript source and uses the `sqliteTable()` calls to understand table structure.

**How to avoid:** Always point drizzle.config schema at `.ts` source files. Include `src/` in npm `files` array for `@minion-stack/db`.

### Pitfall 2: Breaking Both Runners Active Simultaneously

**What goes wrong:** Hub's `db:push` and meta-repo's `db:push` both point at the same live Turso DB and both run at the same time.

**Why it happens:** Incomplete Step 1 ŌåÆ Step 2 transition.

**How to avoid:** Step 2 is blocked by explicit removal of `db:push` from hub's `package.json` scripts before meta-repo takes ownership. There is no automated runner; both are manual CLI commands. Risk is low (human must invoke). Staging dry-run gate (plan 05-04) acts as circuit breaker.

### Pitfall 3: pnpm Workspace Not Recognizing packages/db

**What goes wrong:** `pnpm install` at meta-repo root doesn't recognize `packages/db` as a workspace package.

**Why it happens:** `pnpm-workspace.yaml` uses `packages: ["packages/*"]` glob ŌĆö this auto-includes any directory with a `package.json`. Non-issue if `packages/db/package.json` is created correctly. [VERIFIED: pnpm-workspace.yaml inspection]

**How to avoid:** Create `packages/db/package.json` with `name: "@minion-stack/db"` before running pnpm install.

### Pitfall 4: Bun Consumer Cannot Resolve Workspace Packages from pnpm Workspace

**What goes wrong:** `minion_hub` uses bun, not pnpm. After `@minion-stack/db` is published to npm and `bun add @minion-stack/db` is run in hub, this is straightforward. But during development (pre-publish), bun cannot use `workspace:*` resolution from the pnpm meta-repo workspace.

**Why it happens:** Hub is an independent git repo with its own bun installation ŌĆö it does not participate in the meta-repo pnpm workspace.

**How to avoid:** Plan 05-01 publishes `@minion-stack/db@0.1.0` to npm before hub and site update their imports. Hub and site always consume from npm, never from workspace. This matches the Phase 4 pattern. [VERIFIED: Phase 4 SUMMARY.md precedent]

### Pitfall 5: tsconfig strict mode in @minion-stack/db

**What goes wrong:** `library.json` base tsconfig enables `noUncheckedIndexedAccess: true` (inherited from `base.json`). Hub schema files were written with `minion_hub`'s transitional tsconfig (which has `noUncheckedIndexedAccess: false`). Some array indexed accesses in schema files may fail.

**Why it happens:** Same issue as Phase 4's `_formatTs: hm[1]` fix. Phase 4 fixed it in shared/src/utils/text.ts.

**How to avoid:** Run `pnpm --filter @minion-stack/db build` immediately after copying schema files. Any type errors in strict mode must be fixed before publishing. Look for array-indexed access patterns like `arr[0]` used as non-nullable.
